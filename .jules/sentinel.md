## 2026-03-24 - Unvalidated Editor Process Query via Shell Commands
**Vulnerability:** The `editor status` action used raw shell commands (`pgrep` and `tasklist`) through `execFile` to find Godot processes. It parsed the untrusted string output using regular expressions to extract PIDs.
**Learning:** Using system tools to globally query processes and manually parsing string output is brittle and exposes the system to potential injection or parsing bugs, especially if malicious process names are introduced or if regexes are loosely bounded.
**Prevention:** Instead of querying global system state via shell commands, track process lifecycles internally (e.g., `config.activePids`) when launched by the tool. To verify their existence, use safe OS-level APIs like `process.kill(pid, 0)`, which tests process existence synchronously without relying on parsing string output or shelling out to external commands.

## 2026-04-03 - Unbounded Input Length DoS via String Similarity
**Vulnerability:** The `findClosestMatch` function calculated string similarity (Sørensen–Dice bigrams) on untrusted user inputs (like MCP tool action names) without length limits. A maliciously crafted, extremely long string would cause excessive memory allocation and CPU exhaustion during bigram generation. Additionally, logging or returning this string in error messages via `throwUnknownAction` caused log bloat.
**Learning:** Even utility functions used purely for generating helpful error messages (like "Did you mean...?") become DoS vectors if they process unbounded user input using (N)$ or (N^2)$ algorithms.
**Prevention:** Always enforce strict length limits on untrusted input strings before processing them in expensive operations or embedding them in formatted error messages. For action names, truncating to a safe bound (e.g., 100 characters) is sufficient.
